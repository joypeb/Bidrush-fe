# BidRush Frontend UI/UX Guide

본 문서는 `docs/design.md`의 MVP 설계를 바탕으로 BidRush 프론트엔드를 Next.js로 개발하기 위한 UI/UX, 디자인 시스템, 화면 설계, 구현 기준을 정의한다.

- 기준 문서: `docs/design.md`
- 프론트엔드 1차 기준: Next.js App Router
- 대상 제품: 실시간 경매 기반 중고거래 앱
- 최종 수정일: 2026-04-30

---

## 1. 제품 경험 원칙

### 1.1 핵심 경험

BidRush의 첫 번째 제품 경험은 "실시간 경매가 공정하게 진행되고 있다"는 신뢰다. 사용자는 현재가, 최고 입찰자, 남은 시간, 내 입찰 상태, 연결 상태를 즉시 이해할 수 있어야 한다.

주요 UX 목표는 다음과 같다.

1. **빠른 판단**
   - 사용자는 목록에서 경매 상태, 남은 시간, 현재가, 입찰 열기를 빠르게 판단해야 한다.
   - 카드 한 장만 보고도 "지금 들어가야 하는 경매인지" 알 수 있어야 한다.

2. **실시간 신뢰**
   - 경매방에서는 서버가 권위 있는 상태를 가진다는 점을 UI가 명확히 보여줘야 한다.
   - 낙관적 UI는 사용할 수 있지만, 서버 확정 전 상태와 서버 확정 상태를 시각적으로 구분해야 한다.

3. **입찰 집중**
   - 경매방의 중심은 상품 설명이 아니라 현재가, 남은 시간, 입찰 행동이다.
   - 채팅은 상호작용을 강화하지만 입찰 행동을 방해하면 안 된다.

4. **마감 긴장감**
   - 종료 임박, 자동 연장, 최고가 갱신은 즉시 느껴져야 한다.
   - 단, 도박성 과장, 과도한 애니메이션, 불필요한 압박 문구는 피한다.

5. **판매 등록 단순성**
   - 판매자는 사진, 제목, 설명, 시작가, 종료 시간을 입력하면 빠르게 경매를 시작할 수 있어야 한다.
   - 고급 설정은 MVP에서 숨기거나 제외한다.

### 1.2 UX 톤

BidRush는 중고거래의 생활감과 실시간 경매의 속도를 함께 가져야 한다.

- 차분한 기본 화면, 강한 실시간 상태 표현
- 가격과 시간은 크고 명확하게
- 판매 글은 친근하고 읽기 쉽게
- 오류와 제한은 짧고 구체적으로
- 경매 상태는 색상만으로 전달하지 않고 텍스트와 아이콘을 함께 사용

금지할 방향:

- 카지노, 게임 머니, 과도한 승부 자극처럼 보이는 표현
- 어두운 차트 앱처럼 지나치게 금융 서비스처럼 보이는 표현
- 마케팅 랜딩 페이지 중심 구성
- 실제 사용 화면보다 홍보용 히어로가 먼저 나오는 구성

## 2. 사용자와 핵심 작업

### 2.1 구매자

구매자는 경매 목록을 탐색하고, 경매방에 입장하고, 실시간 입찰과 채팅에 참여하고, 낙찰 결과를 확인한다.

구매자의 핵심 질문:

- 지금 진행 중인 경매인가?
- 현재 가격이 얼마인가?
- 남은 시간이 충분한가?
- 내가 최고 입찰자인가?
- 내 입찰이 서버에 확정됐는가?
- 경매가 연장됐는가?
- 낙찰 여부는 무엇인가?

### 2.2 판매자

판매자는 물품 사진과 정보를 등록하고 경매를 시작한다.

판매자의 핵심 질문:

- 어떤 정보만 입력하면 경매를 시작할 수 있는가?
- 사진은 정상적으로 올라갔는가?
- 시작가와 종료 시간이 유효한가?
- 경매가 시작된 뒤 사용자가 보고 있는 화면은 어떤 모습인가?

### 2.3 공통 사용자

공통 사용자에게 중요한 것은 인증, 알림, 연결 상태, 오류 복구다.

- 로그인 상태가 필요한 행동은 행동 직전에 부드럽게 안내한다.
- WebSocket 연결 끊김은 숨기지 않는다.
- 서버 오류는 추상적인 "실패했습니다" 대신 사용자가 다음에 할 일을 알려준다.

## 3. 정보 구조와 라우팅

Next.js App Router 기준으로 아래 구조를 우선 사용한다.

```text
app/
  layout.tsx
  page.tsx
  auctions/
    page.tsx
    [auctionId]/
      page.tsx
  sell/
    new/
      page.tsx
  auth/
    login/
      page.tsx
    signup/
      page.tsx
  me/
    page.tsx
  api/
    auth/
      route.ts
```

### 3.1 주요 라우트

| Route | 목적 | 우선순위 |
|:---|:---|:---|
| `/` | 진행 중인 경매 목록. MVP 첫 화면 | 필수 |
| `/auctions` | 경매 목록. `/`와 동일하거나 필터 확장용 | 필수 |
| `/auctions/[auctionId]` | 실시간 경매방, 입찰, 채팅 | 필수 |
| `/sell/new` | 판매 등록 및 경매 생성 | 필수 |
| `/auth/login` | 로그인 | 필수 |
| `/auth/signup` | 회원가입 | 필수 |
| `/me` | 내 입찰, 판매 경매, 낙찰 결과 | 2차 |

### 3.2 네비게이션

모바일:

- 하단 탭 네비게이션을 기본으로 한다.
- 탭: `경매`, `판매`, `내 정보`
- 경매방에서는 하단 탭보다 입찰 패널이 우선이다. 필요하면 탭을 숨긴다.

데스크톱:

- 상단 바를 기본으로 한다.
- 좌측에는 서비스명, 중앙에는 주요 탭, 우측에는 로그인/프로필을 둔다.
- 경매방은 화면 폭에 따라 2컬럼 또는 3컬럼 레이아웃을 사용한다.

전역 네비게이션은 현재 화면의 핵심 행동을 밀어내면 안 된다.

### 3.3 프론트엔드 앱 배치

백엔드 저장소 안에서 먼저 시작한다면 `frontend/` 디렉터리에 Next.js 앱을 둔다.

```text
bidrush-be/
  docs/
  rules/
  src/
  frontend/
    app/
    src/
    package.json
```

별도 저장소로 분리하더라도 라우팅, 디자인 토큰, 컴포넌트 구조는 이 문서를 기준으로 유지한다. API 계약은 백엔드 OpenAPI 문서를 기준으로 동기화한다.

## 4. 화면별 UX 설계

### 4.1 경매 목록 화면

목적: 사용자가 참여할 경매를 빠르게 고른다.

구성:

- 상단: 검색, 카테고리 필터, 상태 필터
- 본문: 경매 카드 그리드 또는 리스트
- 카드: 이미지, 제목, 현재가, 남은 시간, 입찰 수, 상태
- 하단 또는 FAB: 판매 등록 진입

카드에 반드시 표시할 정보:

- 상품 대표 이미지
- 상품명
- 현재 최고가
- 남은 시간
- 상태: 진행 중, 종료 임박, 종료, 낙찰 처리 중
- 입찰 수 또는 참여자 수

정렬 기본값:

1. 종료 임박 경매
2. 새로 등록된 경매
3. 입찰이 활발한 경매

필터 MVP:

- 전체
- 진행 중
- 종료 임박
- 종료됨

빈 상태:

- 진행 중인 경매가 없을 때는 판매 등록 행동을 제안한다.
- 문구 예시: `지금 진행 중인 경매가 없습니다. 첫 경매를 등록해보세요.`

로딩 상태:

- 카드 스켈레톤을 사용한다.
- 이미지 영역, 제목, 가격, 시간 위치를 실제 카드와 같은 크기로 잡는다.

### 4.2 경매방 화면

목적: 사용자가 실시간으로 상태를 보고 안전하게 입찰한다.

모바일 레이아웃:

```text
[상단 바: 뒤로가기, 경매 상태, 연결 상태]
[상품 이미지]
[상품 제목, 판매자, 설명 접기/펼치기]
[현재가 + 남은 시간]
[최근 입찰 로그 또는 최고 입찰자]
[채팅]
[고정 입찰 패널]
```

데스크톱 레이아웃:

```text
[상단 바]
[좌측: 상품 이미지, 설명, 판매자 정보]
[중앙: 현재가, 카운트다운, 입찰 패널, 입찰 로그]
[우측: 채팅]
```

경매방에서 가장 큰 시각 요소:

1. 현재가
2. 남은 시간
3. 입찰 버튼
4. 내 상태

상태 표시:

- `내가 최고 입찰 중`은 성공 상태로 표시한다.
- `다른 사용자가 더 높은 금액을 입찰했습니다`는 경고가 아니라 정보 상태로 표시한다.
- `입찰 처리 중`은 버튼 내부 로딩과 별도 상태 문구를 같이 보여준다.
- 서버 확정 전에는 `확인 중`으로 표시한다.

입찰 입력:

- 기본은 빠른 금액 버튼을 제공한다.
- 직접 입력은 보조로 제공한다.
- 최소 입찰 단위를 UI에 표시한다.
- 현재가 이하 금액은 클라이언트에서 즉시 막되, 최종 검증은 서버 응답을 따른다.

빠른 금액 버튼 예시:

- `+1,000`
- `+5,000`
- `+10,000`
- `직접 입력`

입찰 버튼 문구:

- 기본: `입찰하기`
- 처리 중: `입찰 확인 중`
- 최고 입찰 중: `최고 입찰 유지`
- 종료됨: `경매 종료`
- 로그인 필요: `로그인 후 입찰`

경매 종료:

- 종료 직후에는 서버 최종 결과 확정 전까지 `낙찰 결과 확인 중` 상태를 보여준다.
- 낙찰자에게는 `낙찰되었습니다`와 다음 행동을 보여준다.
- 비낙찰자에게는 `경매가 종료되었습니다`와 비슷한 경매 탐색 행동을 보여준다.

### 4.3 채팅

목적: 물품별 커뮤니케이션과 실시간 이벤트 이해를 돕는다.

채팅 구성:

- 일반 메시지
- 시스템 메시지
- 입찰 이벤트 메시지
- 자동 연장 메시지
- 경매 종료 메시지

시스템 메시지 예시:

- `새 입찰이 들어왔습니다. 현재가 42,000원`
- `종료 30초 전 입찰로 경매 시간이 20초 연장되었습니다.`
- `경매가 종료되었습니다. 낙찰 결과를 확인 중입니다.`

채팅 UX 규칙:

- 새 메시지가 도착해도 사용자가 과거 메시지를 보고 있으면 자동 스크롤하지 않는다.
- 대신 `새 메시지` 버튼을 표시한다.
- 시스템 메시지는 일반 메시지보다 작은 크기와 중립 배경을 사용한다.
- 입찰 이벤트는 가격과 시간을 포함한다.
- 욕설, 신고, 차단은 MVP 이후 기능으로 두되 UI 확장 위치를 남긴다.

### 4.4 판매 등록 화면

목적: 최소 입력으로 경매를 시작한다.

필수 입력:

- 사진 1장 이상
- 상품명
- 상품 설명
- 시작가
- 경매 종료 시간
- 카테고리

입력 순서:

1. 사진
2. 상품명
3. 설명
4. 시작가
5. 종료 시간
6. 카테고리
7. 미리보기
8. 경매 시작

사진 업로드:

- 첫 사진을 대표 이미지로 사용한다.
- 업로드 중, 업로드 실패, 재시도 상태를 각 이미지 슬롯에 표시한다.
- 이미지가 너무 크거나 지원하지 않는 형식이면 즉시 안내한다.

가격 입력:

- 숫자 키패드가 열리도록 `inputMode="numeric"`을 사용한다.
- 천 단위 구분을 화면에 표시한다.
- 실제 전송 값은 숫자 또는 서버 계약에 맞춘 문자열로 보낸다.

종료 시간:

- MVP에서는 빠른 선택을 우선한다.
- 예: `30분`, `1시간`, `3시간`, `오늘 자정`
- 서버는 UTC+00:00 기준 `Instant`를 사용하므로 프론트엔드는 사용자 로컬 시간으로 보여주고 API 전송 시 ISO-8601 UTC 문자열로 변환한다.

제출 전 미리보기:

- 목록 카드에서 보일 모습
- 경매방 상단에서 보일 제목과 이미지
- 시작가와 종료 예정 시간

### 4.5 인증 화면

목적: 입찰과 판매 전 필요한 인증을 빠르게 완료한다.

로그인 유도 원칙:

- 목록 탐색은 비로그인으로 허용할 수 있다.
- 입찰, 채팅 전송, 판매 등록 제출 시점에는 로그인이 필요하다.
- 사용자가 하던 행동을 잃지 않도록 로그인 후 원래 화면으로 돌아온다.

문구:

- `입찰하려면 로그인이 필요합니다.`
- `로그인하면 이 경매방으로 돌아옵니다.`

### 4.6 내 정보 화면

MVP 2차 기능으로 두되 정보 구조는 미리 잡는다.

탭:

- 내 입찰
- 판매 중
- 낙찰 결과

각 항목:

- 상품 이미지
- 상품명
- 현재가 또는 낙찰가
- 경매 상태
- 남은 시간 또는 종료 시간
- 다음 행동

## 5. 디자인 시스템

### 5.1 디자인 방향

BidRush의 UI는 "조용한 거래 앱 위에 실시간 경매 상태가 선명하게 올라오는" 방향을 따른다.

- 기본 화면은 흰색과 중립색 중심
- 실시간 이벤트는 제한된 강조색 사용
- 가격, 시간, 상태는 높은 대비
- 카드와 패널은 작은 radius와 얇은 border
- 장식용 그라디언트, 큰 배경 도형, 의미 없는 일러스트는 사용하지 않는다.

### 5.2 색상 토큰

색상은 의미 기반 토큰으로 사용한다. 컴포넌트에서 원시 색상값을 직접 쓰지 않는다.

```css
:root {
  --color-bg: #f7f8fa;
  --color-surface: #ffffff;
```

## 6. 컴포넌트 가이드

### 6.1 AuctionCard

목적: 목록에서 경매 참여 여부를 빠르게 판단하게 한다.

필수 props:

- `title`
- `imageUrl`
- `currentPrice`
- `bidCount`
- `endsAt`
- `status`
- `isEndingSoon`

표시 규칙:

- 현재가는 제목보다 시각적으로 강해야 한다.
- 남은 시간은 오른쪽 상단 또는 가격 근처에 둔다.
- 종료 임박 상태는 배지와 카운트다운 색상을 함께 바꾼다.

### 6.2 CountdownTimer

목적: 서버 기준 종료 시간을 사용자에게 안정적으로 보여준다.

규칙:

- 서버 시간을 기준으로 offset을 계산한다.
- 클라이언트 로컬 시계만 믿지 않는다.
- 60초 이하부터 초 단위 강조를 시작한다.
- 10초 이하에서는 위험색과 진동감 있는 애니메이션을 짧게 사용한다.
- 자동 연장 이벤트를 받으면 남은 시간을 즉시 갱신하고 시스템 메시지를 추가한다.

표시 예:

- `12분 04초`
- `58초`
- `종료`
- `연장됨 +20초`

### 6.3 BidPanel

목적: 빠르고 안전한 입찰 행동을 제공한다.

상태:

- `idle`
- `editing`
- `submitting`
- `accepted`
- `rejected`
- `outbid`
- `closed`
- `loginRequired`

규칙:

- `submitting` 중에는 중복 제출을 막는다.
- 서버 응답이 오기 전까지 성공처럼 확정 표현하지 않는다.
- 실패 시 버튼 위에 짧은 이유를 표시한다.
- `409 Conflict` 계열 응답은 최신 현재가를 반영한 뒤 다시 입찰할 수 있게 한다.

실패 문구 예:

- `현재가보다 높은 금액을 입력해주세요.`
- `방금 더 높은 입찰이 들어왔습니다. 현재가를 확인해주세요.`
- `경매가 종료되어 입찰할 수 없습니다.`
- `연결이 불안정합니다. 잠시 후 다시 시도해주세요.`

### 6.4 PriceTicker

목적: 현재가 변경을 즉시 인지하게 한다.

규칙:

- 가격이 오르면 600ms 이하의 짧은 강조 효과를 사용한다.
- 강조 효과는 배경색 flash 또는 border flash 정도로 제한한다.
- 가격 숫자 자체가 흔들리거나 카드 크기를 바꾸면 안 된다.
- 가격은 항상 통화 포맷으로 표시한다.

예:

```ts
new Intl.NumberFormat("ko-KR", {
  style: "currency",
  currency: "KRW",
  maximumFractionDigits: 0,
}).format(price)
```

### 6.5 ChatPanel

목적: 사용자 대화와 경매 이벤트를 함께 보여준다.

규칙:

- 일반 메시지와 시스템 메시지는 스타일을 구분한다.
- 입력창은 모바일에서 입찰 패널과 겹치지 않아야 한다.
- 전송 실패 메시지는 재시도 버튼을 제공한다.
- 새 메시지 알림은 하단 floating 버튼으로 제공한다.

### 6.6 ConnectionStatus

목적: 실시간 연결 상태를 숨기지 않는다.

상태:

## 7. 실시간 UX 정책

### 7.1 서버 권위 원칙

`docs/design.md`의 원칙처럼 모든 경매 상태 결정권은 서버에 있다.

프론트엔드는 다음 상태를 구분해야 한다.

| 상태 | 의미 | UI 표현 |
|:---|:---|:---|
| Local Draft | 사용자가 입력 중 | 일반 입력 상태 |
| Pending | 서버 확인 대기 | 버튼 로딩, `확인 중` |
| Accepted | 서버 확정 | 현재가 갱신, 성공 상태 |
| Rejected | 서버 거절 | 사유 표시, 최신 상태 반영 |
| Synced | 서버 최신 상태와 일치 | 정상 상태 |

### 7.2 WebSocket 이벤트 처리

예상 이벤트:

- `auction.snapshot`
- `auction.bid.accepted`
- `auction.bid.rejected`
- `auction.outbid`
- `auction.extended`
- `auction.closed`
- `chat.message.created`
- `chat.system.created`

이벤트 처리 규칙:

- 모든 이벤트에는 `auctionId`, `eventId`, `occurredAt`이 있어야 한다고 가정하고 UI 중복 반영을 막는다.
- 순서가 뒤섞일 수 있으므로 서버 timestamp와 version이 있으면 이를 기준으로 정렬한다.
- 알 수 없는 이벤트는 무시하되 개발 환경에서 로깅한다.
- 경매방 진입 시 snapshot을 먼저 받고 이후 delta 이벤트를 적용한다.

### 7.3 자동 연장 UX

정책:

- 종료 30초 전 입찰 발생 시 종료 시간 20초 연장
- 최대 3회 연장

UI 표시:

- 카운트다운 옆에 `연장 1/3` 표시
- 연장 발생 시 시스템 메시지 추가
- 타이머 영역에 `+20초 연장` short toast 표시
- 연장 횟수가 최대에 도달하면 `마지막 연장` 표시

### 7.4 연결 끊김

연결이 끊기면:

- 상단 또는 경매방 상태 영역에 표시한다.
- 입찰과 채팅 전송을 잠시 막는다.
- 입력 중인 채팅 메시지는 유지한다.
- 재연결 중임을 보여준다.
- 재연결 성공 후 최신 상태를 다시 동기화한다.

사용자 문구:

- `실시간 연결이 끊겼습니다. 다시 연결 중입니다.`
- `연결이 복구되었습니다. 최신 경매 상태를 확인했습니다.`

## 8. 상태, 오류, 빈 화면

### 8.1 로딩

- 첫 화면 로딩은 skeleton을 사용한다.
- 경매방 초기 진입은 상단, 가격, 채팅 영역 skeleton을 분리한다.
- 입찰 버튼 로딩은 전체 화면을 막지 않는다.

### 8.2 오류

REST API 오류는 Problem Details 형식을 기준으로 처리한다.

예상 필드:

- `type`
- `title`
- `status`
- `detail`
- `instance`
- `traceId`
- `errors` 또는 `invalidParams`

UI 매핑:

| HTTP | 상황 | UI |
|:---|:---|:---|
| 400 | 입력 오류 | 필드 오류 또는 상단 오류 |
| 401 | 인증 필요 | 로그인 유도 |
| 403 | 권한 없음 | 접근 불가 안내 |
| 404 | 경매 없음 | 빈 상태 + 목록 이동 |
| 409 | 입찰 충돌 | 최신 현재가 반영 + 재입찰 |
| 429 | 요청 과다 | 재시도 가능 시간 표시 |
| 500 | 서버 오류 | 짧은 안내 + 재시도 |

### 8.3 빈 화면

빈 화면은 다음 행동을 제공해야 한다.

- 경매 없음: `판매 등록`
- 검색 결과 없음: `필터 초기화`
- 내 입찰 없음: `경매 둘러보기`
- 채팅 없음: `첫 메시지를 보내보세요`

## 10. 접근성

필수 기준:

- 모든 버튼은 키보드로 접근 가능해야 한다.
- focus style을 제거하지 않는다.
- icon-only 버튼은 `aria-label`을 가진다.
- 가격과 시간 갱신 영역은 필요한 경우 `aria-live="polite"`를 사용한다.
- 입찰 성공/실패는 toast만으로 전달하지 않고 화면 내 상태도 갱신한다.
- 색상만으로 종료 임박, 성공, 실패를 전달하지 않는다.

실시간 영역:

- 너무 잦은 `aria-live` 알림은 사용자를 방해할 수 있다.
- 현재가와 남은 시간은 핵심 변화만 알린다.
- 채팅 새 메시지는 사용자가 채팅 패널에 focus 중일 때만 적절히 알린다.

## 12. Next.js 구현 기준

### 12.1 기술 선택

권장 스택:

- Next.js App Router
- TypeScript
- React Server Components
- TanStack Query 또는 SWR
- Zustand 또는 React Context for lightweight client state
- STOMP WebSocket client
- Zod for runtime schema validation
- Tailwind CSS 또는 CSS Modules
- lucide-react

선택 기준:

- 서버에서 가져올 수 있는 정적/초기 데이터는 Server Component에서 처리한다.
- 실시간 상태와 사용자 입력은 Client Component에서 처리한다.
- 경매방은 WebSocket 때문에 Client Component 비중이 높아질 수 있다.

### 12.2 데이터 패칭

목록:

- SSR 또는 RSC로 첫 목록을 가져온다.
- 필터 변경은 URL query에 반영한다.
- pagination 또는 cursor를 API 계약에 맞춘다.

경매방:

1. REST로 snapshot 조회
2. WebSocket 연결
3. 이벤트 구독
4. snapshot version 이후 이벤트만 반영
5. 연결 복구 시 snapshot 재조회

판매 등록:

- 이미지 업로드는 presigned URL 또는 백엔드 업로드 API 계약에 맞춘다.
- 제출 전 클라이언트 검증을 수행한다.
- 서버 검증 실패는 필드 오류로 매핑한다.

### 12.3 상태 관리

서버 상태:

- 경매 목록
- 경매 snapshot
- 사용자 프로필
- 내 입찰/판매 목록

클라이언트 상태:

- 현재 입력 중인 입찰 금액
- 채팅 입력값
- WebSocket 연결 상태
- 낙관적 pending 이벤트
- 모달, 하단 시트, toast

경매방 상태는 reducer 형태를 권장한다.

```ts
type AuctionRoomState = {
  auctionId: string;
  currentPrice: number;
  bidCount: number;
  endsAt: string;
  status: "OPEN" | "CLOSED";
  extensionCount: number;
  connectionStatus: "connected" | "connecting" | "reconnecting" | "offline";
  pendingBidId?: string;
};
```

### 12.5 API 클라이언트

규칙:

- DTO 타입은 OpenAPI 생성 타입을 우선한다.
- 직접 타입을 만들 경우 API 계약 변경 시 한 곳에서 수정할 수 있게 `features/*/schemas.ts`에 모은다.
- 날짜는 문자열로 받고 화면 직전에 변환한다.
- 금액 포맷은 공통 `formatCurrency`를 사용한다.

프론트엔드가 백엔드에 우선 요청해야 하는 계약:

| 리소스 | 필요 데이터 | UX 영향 |
|:---|:---|:---|
| 경매 목록 | 제목, 대표 이미지, 현재가, 종료 시간, 상태, 입찰 수 | 목록 카드와 정렬 |
| 경매 snapshot | 현재가, 최고 입찰자, 종료 시간, 연장 횟수, 상태, 서버 시간 | 경매방 초기 상태 |
| 입찰 요청 | 입찰 금액, idempotency key | 중복 입찰 방지 |
| 입찰 응답 | accepted/rejected, 최신 현재가, 사유 코드 | BidPanel 상태 전환 |
| 채팅 목록 | 메시지, 작성자, 시스템 여부, 생성 시간 | ChatPanel 초기 렌더 |
| 이미지 업로드 | 업로드 URL 또는 업로드 결과 URL | 판매 등록 |
| 인증 | access token 갱신 정책, 사용자 식별자 | 로그인 유지와 WebSocket 인증 |

입찰 실패 reason code는 UI 문구와 직접 연결되므로 안정적인 enum으로 제공받는다.

### 12.6 WebSocket 클라이언트

규칙:

- 연결 생성, 재연결, 구독 해제는 `lib/realtime.ts`로 모은다.
- 컴포넌트에서 STOMP client를 직접 다루지 않는다.
- 경매방 unmount 시 반드시 unsubscribe한다.
- 중복 구독을 막는다.

### 12.7 환경 변수

예상:

```text
NEXT_PUBLIC_API_BASE_URL=http://localhost:8080
NEXT_PUBLIC_WS_BASE_URL=ws://localhost:8080/ws
```

규칙:

- 브라우저에 노출되는 값만 `NEXT_PUBLIC_`을 붙인다.
- 토큰, 서버 비밀키, S3 secret은 프론트엔드 환경 변수에 두지 않는다.

## 14. 테스트와 QA 기준

### 14.1 단위 테스트

대상:

- 가격 포맷
- 남은 시간 계산
- 서버 시간 offset 계산
- room reducer 이벤트 적용
- 입찰 금액 검증

### 14.2 컴포넌트 테스트

대상:

- AuctionCard 상태 표시
- BidPanel 상태별 버튼 문구와 disabled 처리
- CountdownTimer 종료/연장 표시
- ChatPanel 새 메시지 표시
- FormField 오류 표시

### 14.3 E2E 테스트

필수 시나리오:

1. 경매 목록 조회 후 경매방 입장
2. 로그인 필요 상태에서 입찰 시 로그인 유도
3. 입찰 성공 후 현재가 갱신
4. 더 높은 입찰 이벤트 수신 시 outbid 표시
5. 종료 임박 입찰로 자동 연장 표시
6. WebSocket 연결 끊김과 재연결
7. 판매 등록 폼 검증 실패
8. 판매 등록 성공 후 경매방 이동

### 14.4 수동 QA 체크리스트

- 모바일 360px 너비에서 텍스트가 버튼 밖으로 넘치지 않는다.
- 가격 갱신 시 레이아웃이 흔들리지 않는다.
- 채팅 입력창과 입찰 패널이 겹치지 않는다.
- 경매방에서 뒤로가기 후 WebSocket 구독이 해제된다.
- 재연결 후 최신 현재가와 남은 시간이 맞다.
- 색상 없이도 상태를 이해할 수 있다.

## 16. 개발 시작 순서

1. Next.js 앱 생성
2. 디자인 토큰과 기본 레이아웃 구성
3. API client와 공통 formatter 작성
4. 경매 목록 화면 구현
5. 경매방 snapshot UI 구현
6. WebSocket 연결과 이벤트 reducer 구현
7. BidPanel 구현
8. ChatPanel 구현
9. 판매 등록 폼 구현
10. 인증 흐름 연결
11. E2E 테스트 작성

첫 개발 마일스톤은 "목록에서 경매방에 들어가 현재가와 남은 시간을 보고, mock WebSocket 이벤트로 가격이 갱신되는 화면"이다. 이 마일스톤이 잡히면 BidRush의 핵심 감각을 빠르게 검증할 수 있다.

## 17. 최종 품질 기준

프론트엔드 MVP는 아래 질문에 모두 "예"라고 답할 수 있어야 한다.

- 사용자는 현재가와 남은 시간을 1초 안에 이해하는가?
- 사용자는 내 입찰이 확정됐는지 대기 중인지 구분할 수 있는가?
- 자동 연장 이벤트가 명확하게 보이는가?
- 연결 끊김 상태에서 사용자가 잘못된 입찰을 할 수 없게 막는가?
- 모바일에서 입찰 버튼이 항상 안전하게 접근 가능한가?
- 판매자는 1분 안에 경매 등록을 완료할 수 있는가?
- 서버 오류와 검증 오류가 사용자가 이해할 수 있는 말로 표시되는가?
- 화면이 디자인 장식보다 실제 경매 행동에 집중하는가?

이 기준을 만족하면 Next.js 프론트엔드의 첫 구현 방향은 충분히 단단하다.
